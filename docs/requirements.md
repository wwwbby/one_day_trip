# Daytrip Planner 详细需求文档

版本：0.1  
日期：2026-07-09  
项目目录：`work/daytrip-planner`

## 1. 项目概述

Daytrip Planner 是一个面向日本旅行、动画巡礼和城市一日行程安排的地图规划工具。用户可以创建多个“一日规划”，在地图上添加起点、终点、定点、普通点和巡礼点，并基于指定日期与地点停留时间自动规划一天内的访问顺序和公共交通路线。

项目参考 Anitabi 的点位浏览与巡礼体验，但核心目标不是复刻 Anitabi，而是将“作品点位导入 + 地图点位管理 + 一日路线规划 + 服务端持久化”整合成一个可在电脑和手机上同步使用的行程工具。

## 2. 产品目标

### 2.1 核心目标

- 支持用户创建、保存、编辑多个一日行程规划。
- 支持从地图、地点搜索、坐标、Google Maps 链接、KML、Anitabi 作品导入地点。
- 支持按日期、起点、终点、定点、停留时间自动规划一天的访问顺序。
- 支持日本公共交通路线查询，公共交通包含 JR、地下铁、私铁、巴士等 transit 总类，而不是只包含公交车。
- 支持 Anitabi 巡礼点图片悬浮预览和大图查看。
- 支持服务端持久化，使电脑和手机访问同一部署地址时看到同一份规划。

### 2.2 设计原则

- 地图是主工作区，规划操作应围绕地图和地点列表展开。
- 自动规划以“当天可执行”为优先，而不是只追求直线距离最短。
- 定点是硬约束：必须优先保证在定点开始时间前到达。
- 超过 1km 的相邻地点优先查询公共交通，1km 内优先步行。
- 不提供驾车和骑行作为主要规划方式。
- API 不可用时应有兜底方案，而不是让用户卡死。
- 用户已有规划数据不能因为刷新、部署或切换设备而丢失。

## 3. 用户与使用场景

### 3.1 目标用户

- 去日本旅行并希望按天规划路线的用户。
- 动画巡礼用户，需要从 Anitabi 导入作品点位并筛选一天内可去的地点。
- 需要在手机上查看电脑端已规划路线的用户。
- 对公共交通路线、换乘时间和行程可达性比较敏感的用户。

### 3.2 典型场景

1. 用户计划 7 月 19 日进行一天巡礼，从酒店出发，晚上 17:30 必须到达演出场馆。
2. 用户搜索作品名，从 Anitabi 导入该作品所有截图点位。
3. 用户添加酒店作为起点和终点，添加演出场馆作为定点。
4. 用户点击“自动规划路线”。
5. 系统将能赶在定点前访问的点排在定点前，塞不下的普通点或巡礼点挪到定点后。
6. 如果即使直接去定点也无法准时到达，系统标记该定点为当天无法合理安排。
7. 用户保存规划，在手机浏览器打开同一部署地址查看路线。

## 4. 核心概念

### 4.1 一日规划 DayPlan

一日规划是一组在同一天执行的地点、路线和排程数据。每个规划包含：

- 规划名称。
- 行程日期。
- 地点列表。
- 路线规划结果。
- 创建时间和更新时间。
- 服务端同步状态。

### 4.2 地点 Stop

地点是路线规划的基本单元。地点可以来自：

- Google Places 搜索。
- 免费模式下的 Nominatim 搜索。
- 用户点击地图。
- 用户粘贴坐标或 Google Maps 链接。
- KML 导入。
- Anitabi 作品点位导入。

地点基础字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 地点唯一标识 |
| `name` | 地点名称 |
| `address` | 地址或来源描述 |
| `location.lat/lng` | WGS84 坐标 |
| `stayMinutes` | 停留时间，所有点位均可设置 |
| `role` | 地点角色：普通点、巡礼点、定点、终点 |
| `isStart` | 是否为起点 |
| `isEnd` | 是否为终点 |
| `windowStart` | 起点或定点开始时间 |
| `note` | 用户备注 |
| `source` | 数据来源 |
| `imageUrl/thumbnailUrl` | Anitabi 图片 |
| `anitabiPointId/workTitle` | Anitabi 元数据 |

### 4.3 地点类型

| 类型 | 说明 | 时间规则 |
| --- | --- | --- |
| 起点 | 当天出发地点，可以同时作为终点 | 只需要开始时间和停留时间 |
| 终点 | 当天结束地点 | 默认停留时间可为 0，可作为最后目的地 |
| 定点 | 必须在指定开始时间前到达的地点 | 开始时间 + 停留时间，结束时间自动计算 |
| 普通点 | 用户手动添加的一般地点 | 使用停留时间 |
| 巡礼点 | Anitabi 导入或用户设置的巡礼地点 | Anitabi 导入点默认是巡礼点 |

## 5. 功能需求

### 5.1 规划首页

#### FR-001 多规划列表

- 系统必须在首页展示所有已保存的一日规划。
- 每个规划卡片显示日期、名称、地点数量、已规划路线距离、路上时间、保存时间。
- 用户可以进入某个规划详情页。

#### FR-002 创建规划

- 用户可以选择日期并输入规划名称。
- 用户点击新建后进入该规划编辑页。
- 如果用户不输入名称，系统应提供可识别的默认名称。

#### FR-003 删除规划

- 用户可以删除某个规划。
- 删除操作必须同步到服务端。
- 删除失败时必须显示明确错误信息。

#### FR-004 服务端同步提示

- 页面应显示当前是服务端同步还是本地缓存模式。
- 当服务端不可用时，应退回本地缓存并提示用户。

### 5.2 规划编辑页

#### FR-010 地图显示

- 优先使用 Google Maps JavaScript API。
- 没有 Google key 或 Google Maps 加载失败时，自动切换到 Leaflet + OpenStreetMap。
- 地图上必须显示所有地点 marker。
- marker 应使用颜色或样式区分起点、终点、定点、普通点、巡礼点。
- 用户点击某个 marker 或地点卡片时，地图应以该点为中心放大。

#### FR-011 路线显示

- 自动规划成功后，地图必须显示路线。
- 如果路线 API 返回折线坐标，应使用真实折线。
- 如果路线 API 没有返回可用 geometry，应使用规划后的点位坐标连线兜底，避免地图无路线。
- 规划路线后，地图应缩放到路线范围，而不是回到默认城市中心。

#### FR-012 地点列表

- 左侧或下方地点列表应展示所有地点。
- 每个地点卡片显示：
  - 序号或类型标记。
  - 名称。
  - 地址或坐标。
  - 备注。
  - 类型下拉选择。
  - 停留时间输入。
  - 起点或定点开始时间输入。
  - 预计到达/访问/离开时间。
  - 到下一站的路线摘要。
  - 删除按钮。
- 地点卡片内容不能遮挡操作按钮。
- 地点时间和类型标签必须在手机端也完整可读。

#### FR-013 地点编辑

- 用户可以修改地点名称、备注、类型、停留时间和开始时间。
- 修改地点、时间或类型后，已有路线应标记为已过期。
- 已过期路线可以继续显示为参考，但必须提示用户点击自动规划刷新。

#### FR-014 拖拽排序

- 用户可以拖拽地点卡片调整顺序。
- 手动排序后，路线应标记为已过期。

#### FR-015 框选与批量删除

- 用户可以开启框选模式。
- 用户在地图上拖拽矩形区域后，系统选中区域内所有地点。
- 用户可以批量删除选中地点。
- 删除后路线应标记为已过期。

### 5.3 添加地点

#### FR-020 搜索地点

- Google 模式下使用 Google Places Autocomplete。
- 免费地图模式下使用 Nominatim。
- 搜索结果应支持用户点击添加。
- 添加地点时可以指定默认类型。

#### FR-021 地图点击添加

- 用户点击地图空白位置可以添加自定义地点。
- 新地点默认名称应可编辑。

#### FR-022 坐标和链接解析

- 输入框应支持粘贴：
  - `lat,lng` 坐标。
  - Google Maps `@lat,lng` 链接。
  - Google Maps `!3dlat!4dlng` 链接。
  - Google Maps `q=lat,lng` 或 `ll=lat,lng` 链接。

#### FR-023 KML 导入

- 用户可以导入 KML 文件。
- 系统解析 `Placemark` 名称、描述和坐标。
- 无法解析时显示错误。

### 5.4 Anitabi 导入

#### FR-030 作品模糊搜索

- 用户输入作品名后，系统调用后端 Anitabi 搜索代理。
- 搜索结果显示作品标题、中文名、日期、城市、封面、点位数量和样例点位。
- 用户可以从搜索结果中选择导入。

#### FR-031 按 subjectId 导入

- 系统应支持根据 Bangumi subjectId 拉取 Anitabi 点位。
- 导入接口必须通过后端代理，避免前端跨域或接口格式变化直接影响 UI。

#### FR-032 导入点位规则

- Anitabi 导入的所有点位默认类型为巡礼点。
- 默认停留时间为 5 分钟。
- 默认备注为“巡礼”。
- 地址字段应包含作品名、集数、场景时间等可读信息。
- 必须保存原图、缩略图、来源用户、Anitabi point id、作品标题等元数据。

#### FR-033 图片预览

- 地图 marker 悬浮时显示小图预览。
- 左侧地点列表悬浮时不显示图片，避免挡住编辑按钮。
- 用户点击“查看图片”后显示大图弹窗。
- 大图链接裂开时应自动降级到可用图片变体。

### 5.5 自动路线规划

#### FR-040 自动排序

- 用户只需要提供当天起点、若干地点、可选终点和可选定点。
- 系统应自动规划访问顺序。
- 自动排序应优先考虑：
  1. 起点固定在第一站。
  2. 终点固定在最后一站。
  3. 定点必须准时或提前到达。
  4. 相近地点尽量连续访问。
  5. 总行程时间尽量短。

#### FR-041 定点硬约束

- 定点开始时间是硬约束。
- 系统必须优先保证在定点开始时间前到达。
- 如果普通点或巡礼点放在定点前会导致迟到，系统应自动将其挪到该定点之后。
- 如果真实路线 API 返回后仍显示定点迟到，系统应继续移动定点前的可移动点并重新规划。
- 如果已经没有可移动点，或即使直接前往定点仍无法准时，系统应标记该定点为当天无法合理安排。
- 定点停留时间用于计算定点离开时间：`离开时间 = max(实际到达时间, 定点开始时间) + 停留时间`。

#### FR-042 交通方式选择

- 相邻两点直线距离小于等于 1km 时，优先步行。
- 相邻两点直线距离大于 1km 时，优先公共交通。
- 公共交通应包含 JR、地下铁、私铁、巴士等 transit 方式。
- 不应提供驾车和骑行作为自动规划主选项。

#### FR-043 路线查询时间

- 路线查询必须使用用户指定日期。
- 起点路线出发时间为起点开始时间 + 起点停留时间。
- 每一段路线的请求时间必须等于上一地点离开时间。
- 普通点离开时间为到达时间 + 停留时间。
- 定点离开时间为 `max(到达时间, 定点开始时间) + 停留时间`。
- 后续地点开始时间必须等于上一地点结束时间 + 赶路时间。

#### FR-044 路线服务优先级

自动规划路线时，服务端按以下优先级查询：

1. NAVITIME Route(totalnavi)，用于日本公共交通。
2. Google Directions/Routes 服务端 key，用于公共交通或步行兜底。
3. Transitous/MOTIS 免费公共交通实验接口。
4. 直线步行估算兜底。

#### FR-045 API 限流和等待

- NAVITIME 当前限制按 50 次/分钟处理。
- 规划地点多时，不应硬性限制地点数量。
- 前端必须显示加载状态，提示用户等待。
- 服务端按限流节奏排队，直到所有相邻路段规划完成或失败兜底。

#### FR-046 路线结果展示

- 地点之间应显示类似地图 App 的路线卡片。
- 路线卡片包含：
  - 从几号点到几号点。
  - 起终点名称。
  - 总用时。
  - 距离。
  - 换乘次数。
  - 票价。
  - 步行段。
  - 乘坐线路。
  - 方向或 headsign。
  - 起终站。
  - 每一步开始和结束时间。
- 路线卡片第一行不得变成竖排或被挤压。

### 5.6 保存、导出和分享

#### FR-050 自动保存

- 用户修改规划后，系统应自动保存。
- 保存成功显示最近保存时间。
- 服务端保存失败时，应退回本地缓存并提示。

#### FR-051 服务端持久化

- 配置 `DATABASE_URL` 时，规划保存到 Postgres/Neon。
- 未配置数据库的本地开发环境写入 `.data/daytrip-plans.json`。
- Vercel 生产环境必须配置 `DATABASE_URL`，不能依赖本地文件。

#### FR-052 本地缓存

- 浏览器 `localStorage` 仅作为缓存和旧数据迁移来源。
- 服务端首次为空且本地存在旧规划时，应允许迁移上传。

#### FR-053 导出 JSON

- 用户可以导出当前规划 JSON。
- JSON 应包含完整地点、路线和元数据，便于备份。

#### FR-054 复制 Google Maps 链接

- 用户可以复制 Google Maps directions 链接。
- 链接使用当前规划顺序生成 origin、destination 和 waypoints。

## 6. 页面与交互需求

### 6.1 桌面端布局

- 首页以规划列表为主。
- 编辑页应以地图和地点列表为核心。
- 地点列表可滚动，不应因为地图存在而无法滚动。
- 控制按钮应保持可见且不被图片预览遮挡。

### 6.2 手机端布局

- 手机访问时，地图和地点列表应适配窄屏。
- 按钮、输入框、时间、标签不能溢出容器。
- 地图 marker 点击、卡片点击、图片查看应适合触屏。
- 规划列表和编辑页都应能在手机浏览器正常使用。

### 6.3 状态反馈

- 路线规划中显示转圈或加载状态。
- 长路线规划显示预计请求数量或等待提示。
- 路线过期显示醒目提示。
- API 失败显示可理解错误。
- 定点无法满足显示明确原因，不只显示“可能迟到”。

## 9. 路线规划算法需求

### 9.1 输入

- 已排序或未排序的地点列表。
- 起点。
- 可选终点。
- 一个或多个定点。
- 行程日期。
- 起点开始时间。
- 每个地点停留时间。

### 9.2 输出

- 优化后的地点访问顺序。
- 每段路线。
- 总距离。
- 总路上时间。
- 每个地点到达、开始访问、离开时间。
- 定点是否可满足。
- 被挪到定点之后的地点数量。
- 无法满足的定点列表。

### 9.3 排序规则

1. 固定起点。
2. 固定终点。
3. 按定点开始时间从早到晚处理。
4. 尝试在定点前插入普通点或巡礼点。
5. 每次插入前必须判断插入后是否仍能在定点开始前到达。
6. 插不进去的点自动放到定点之后。
7. 定点之后继续按距离和总时间优化。
8. 多个候选顺序按以下优先级比较：
   - 未满足定点数量最少。
   - 定点总迟到时间最少。
   - 总行程耗时更短。
   - 总交通时间更短。
   - 总距离更短。

### 9.4 真实路线后的修正

估算路线可能低估真实公共交通耗时。系统必须在真实路线 API 返回后再次检查定点：

1. 构建真实排程。
2. 如果定点迟到，找到第一个迟到定点。
3. 从该定点前选择可移动的普通点或巡礼点。
4. 将其挪到该定点之后。
5. 重新请求路线。
6. 重复直到：
   - 所有定点准时；或
   - 没有可移动点；或
   - 达到安全重试上限。

### 9.5 无法合理安排

以下情况应标记为无法合理安排：

- 从起点直接去定点也无法准时。
- 定点前没有可移动点，但真实排程仍迟到。
- 多次修复后仍无法满足定点。
- API 长时间失败且无法得到可信路线。

## 10. 环境变量与部署需求

| 变量 | 必填 | 说明 |
| --- | --- | --- |
| `VITE_GOOGLE_MAPS_API_KEY` | 推荐 | 浏览器端 Google Maps JavaScript / Places key |
| `VITE_GOOGLE_MAP_ID` | 可选 | Google Advanced Marker 所需 Map ID |
| `GOOGLE_MAPS_SERVER_KEY` | 可选 | 服务端 Google Directions/Routes key |
| `NAVITIME_RAPIDAPI_KEY` | 推荐 | NAVITIME Route(totalnavi) RapidAPI key |
| `NAVITIME_RAPIDAPI_HOST` | 可选 | 默认 `navitime-route-totalnavi.p.rapidapi.com` |
| `TRANSITOUS_USER_AGENT` | 推荐 | Transitous 请求标识 |
| `DATABASE_URL` | 生产必填 | Neon/Postgres 连接串 |
| `PORT` | 本地可选 | 本地开发端口，默认 5173 |

### 10.1 Vercel 部署

- Root Directory 设为 `work/daytrip-planner`。
- Build Command：`npm run build`。
- Output Directory：`dist`。
- 生产环境必须配置 `DATABASE_URL`。
- API 函数需要允许较长执行时间，以支持长路线规划。
- 不要把服务端 key 放进 `VITE_` 前缀变量。

### 10.2 数据库

- 使用 Neon/Postgres。
- 服务端自动创建 `daytrip_plans` 表。
- 当前所有访问同一部署地址的用户共享同一份规划数据。
- 如果要做多用户隔离，需要增加登录和 `user_id/workspace_id`。

## 11. 非功能需求

### 11.1 可用性

- 没有 Google key 时应用仍可打开。
- 路线 API 失败时应尽量用步行或估算兜底。
- 长时间规划时页面不能卡死。

### 11.2 性能

- 地点几十个时仍能正常编辑和滚动。
- 地图 marker 更新不能导致路线消失。
- 自动规划应显示加载状态，并尽量避免重复请求。

### 11.3 安全

- API key 不得硬编码在前端源码中。
- 服务端 key 不得使用 `VITE_` 前缀。
- 数据库连接串只存在于服务端环境变量。
- 导入外部数据时应校验坐标、文本长度和 JSON 格式。

### 11.4 兼容性

- 桌面 Chrome/Edge/Safari 基本可用。
- 手机浏览器可查看和编辑规划。
- Vercel 生产环境和本地 Express 开发环境行为尽量一致。

## 12. 验收标准

### 12.1 基础流程

- 用户可以新建 7/19 一日规划。
- 用户可以添加酒店为起点。
- 用户可以搜索并添加普通地点。
- 用户可以从 Anitabi 搜索作品并导入巡礼点。
- 用户可以设置一个 17:30 的定点。
- 用户点击自动规划后，系统生成地点顺序、路线卡片和地图路线。
- 用户刷新页面后规划仍存在。
- 用户用手机打开同一地址能看到同一规划。

### 12.2 定点硬约束验收

- 若一个巡礼点会导致 17:30 定点迟到，系统自动将该巡礼点挪到定点之后。
- 若多个点导致迟到，系统持续挪点直到定点准时或无点可挪。
- 若起点直接去定点仍迟到，系统标记该定点无法合理安排。
- 定点停留 180 分钟时，定点之后的路线必须从 `17:30 + 180 分钟` 之后开始查询。

### 12.3 路线时间验收

- 每个地点的开始时间 = 上一地点结束时间 + 上一段赶路时间。
- 普通点结束时间 = 开始时间 + 停留时间。
- 定点结束时间 = `max(到达时间, 定点开始时间) + 停留时间`。
- 公共交通 step 的开始/结束时间与 route leg 时间一致。

### 12.4 地图验收

- 自动规划后地图显示路线。
- API 没有折线时，地图仍用点位连线兜底显示。
- 点击 marker 或地点卡片后，地图以该点为中心放大。
- 地图不会无故缩放到默认城市。

### 12.5 Anitabi 验收

- 输入作品名能展示候选作品。
- 点击导入后点位全部进入列表。
- 导入点位类型均为巡礼点。
- 点击查看图片可打开大图。

## 13. 已知限制

- 免费地图和免费路线服务不适合生产级大流量。
- NAVITIME RapidAPI 有调用频率和套餐限制。
- 多人并发规划时，当前内存级限流不能跨 Vercel 多实例共享。
- 当前没有用户登录，部署地址上的规划对所有访问者共享。
- 自动规划对公共交通真实耗时依赖第三方 API，估算排序仍可能需要真实路线后二次修正。
- 大量地点、多次定点修复可能触发较长等待或 Vercel 函数超时。

## 14. 后续增强

- 用户登录和私有规划。
- 分享只读链接。
